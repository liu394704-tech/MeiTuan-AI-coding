# 「慢病用药小管家」产品级工程设计方案

> 本文件是后续开发的"宪法"。任何代码改动如果与本文不一致，先更新本文，再写代码。

---

## 0. 设计哲学

产品的灵魂不是 CRUD 一堆药品，而是把"问诊→开药→吃药→补药→复诊"这条断裂的慢病链路缝合起来。

主线："**用药闭环**"：每一次用户操作（标记服药、补药、改方案）都回流到统一的 `MedicationEvent` 时间轴，提醒系统、库存系统、AI 模块都从这条时间轴上读数据 → 给反馈。

---

## 1. 产品结构（Information Architecture）

```
慢病用药小管家
├── 用户中心 (User)              —— 谁在用
├── 用药方案 (Medication Plan)   —— 吃什么、怎么吃
├── 今日服药 (Today)             —— 现在该干嘛 ← 首页核心
├── 库存管理 (Inventory)         —— 药还够不够
├── 复诊 (Follow-up)             —— 下次什么时候去医院
├── AI 健康助手 (Insights)       —— 帮我看趋势
└── 历史与报告 (History)         —— 给医生看
```

**关键解耦点**：`Today / Inventory / Follow-up / Insights` 都不直接持有彼此的状态，而是从 `MedicationPlan` + `MedicationEvent` 两张事实表派生。

## 3. 数据模型（JSON Schema）

设计原则：**plan 和 event 分离**（plan 是"应该吃"，event 是"实际吃"）。

```jsonc
// User
{
  "id": "u_001",
  "name": "张大爷",
  "age": 68,
  "gender": "male",
  "conditions": ["hypertension", "type2_diabetes"],
  "allergies": ["penicillin"],
  "createdAt": "2026-01-01T00:00:00Z"
}

// Medication
{
  "id": "med_001",
  "userId": "u_001",
  "name": "氨氯地平片",
  "spec": "5mg/片",
  "form": "tablet",
  "unit": "片",
  "purpose": "降压",
  "prescribedBy": "张医生 / 北京协和",
  "startDate": "2026-04-01",
  "endDate": null
}

// Regimen
{
  "id": "reg_001",
  "medicationId": "med_001",
  "dosage": 1,
  "unit": "片",
  "frequency": {
    "type": "daily",
    "timesPerDay": 2,
    "times": ["08:00", "20:00"],
    "weekdays": [1,2,3,4,5,6,7],
    "intervalDays": 1
  },
  "withFood": "after",
  "notes": "饭后温水送服"
}

// Inventory
{
  "id": "inv_001",
  "medicationId": "med_001",
  "quantity": 28,
  "unit": "片",
  "lowStockThresholdDays": 3,
  "lastRestockedAt": "2026-04-10",
  "lastRestockAmount": 30
}

// MedicationEvent
{
  "id": "evt_20260425_0800_med001",
  "userId": "u_001",

  "regimenId": "reg_001",
  "scheduledAt": "2026-04-25T08:00:00+08:00",
  "status": "pending",
  "takenAt": null,
  "dosage": 1,
  "unit": "片",
  "source": "auto"
}

// FollowUp
{
  "id": "fu_001",
  "userId": "u_001",
  "relatedMedicationIds": ["med_001"],
  "hospital": "北京协和",
  "department": "心内科",
  "doctor": "张医生",
  "scheduledDate": "2026-05-20",
  "reason": "复查血压 + 续方",
  "status": "upcoming",
  "autoGenerated": true
}

// Insight
{
  "id": "ins_001",
  "userId": "u_001",
  "type": "adherence",
  "level": "warn",
  "title": "本周您漏服了 2 次氨氯地平",
  "body": "建议设置睡前提醒或与家人协助。",
  "relatedIds": ["med_001"],
  "createdAt": "..."
}
```

要点：
- `MedicationEvent` 幂等可重放：给定 plan 和日期范围 → 确定性地生成。
- `Inventory.quantity` 在 `event.status` → `taken` 时按 `dosage` 扣减；revert 回滚。
- 时间统一 ISO8601（带时区）。

## 5. 状态管理（Zustand 切片）

```
userStore         → 当前用户、健康档案
medicationStore   → Medication[] + Regimen[]
eventStore        → 今日 + 历史 events
inventoryStore    → Inventory[]
followUpStore     → FollowUp[]
```

派生数据走 hook，不进 store：`useTodayDoses`、`useAdherenceScore`、`useStockForecast`、`useInsights`。

副作用链示例：
```
markDoseAsTaken(eventId)
  ├─► eventStore.update(eventId, {status:'taken', takenAt:now})
  ├─► inventoryStore.decrement(medicationId, dosage)
  └─► insightStore.invalidate()
```

## 6. Mock API 设计

REST 风格，统一 `/api/*`：

| Method | Path | 说明 |
|---|---|---|
| GET | `/api/me` | 当前用户 |
| GET/PUT | `/api/users/:id` | 用户档案 |
| GET/POST | `/api/medications` | 列表 / 新建 |
| GET/PUT/DELETE | `/api/medications/:id` | 详情 / 更新 / 删除 |
| GET | `/api/events?from=&to=` | 区间事件 |
| POST | `/api/events/:id/take|skip|revert` | 状态变更 |
| GET | `/api/inventory` | 库存 |
| POST | `/api/inventory/:medId/restock` | 补药 |
| GET/POST/PUT | `/api/follow-ups` | 复诊 |
| GET | `/api/insights` | AI 洞察 |

两种实现并存，环境变量切换：`VITE_API_MODE=local | msw | remote`。

## 7. AI 模块（规则引擎，可演进）

```
/ai
 ├─ insightEngine.ts
 └─ rules/
     ├─ adherenceRule.ts
     ├─ stockRule.ts
     ├─ riskRule.ts
     └─ tipRule.ts
```

统一规则接口：
```ts
type RuleContext = {
  user: User;
  medications: (Medication & { regimen: Regimen })[];
  events: MedicationEvent[];
  inventory: Inventory[];
  now: Date;
};

interface InsightRule {
  id: string;
  run(ctx: RuleContext): Insight[];
}
```

四条核心规则：
1. **依从性评分**：过去 7 天 `taken/scheduled`，连续漏服加权扣分。
2. **库存预测**：`daysLeft = quantity / 日均消耗`，低于阈值预警 + 加入购药清单。
3. **风险提示**：关键药漏服 / 复诊前库存不足 → danger 卡片。
4. **健康建议**：按 `user.conditions` 匹配静态知识库，每天最多 1 条。

预留 `mode: 'rule' | 'llm' | 'hybrid'` 给未来 LLM 接入。

## 8. UX 关键细节

- **Dashboard 黄金 3 卡**：今日进度环 / 库存预警 / 复诊倒计时
- **添加药品向导 ≤3 步**：基本信息 → 用法（频率模板快捷）→ 库存
- **一键打卡**：≥64px 点击区，2s 内可撤销
- **可视化**：AdherenceRing、StockBar、DoseTimeline
- **适老化**：默认 16px，提供大字模式 18/20px；高对比度配色；图标+文字双标识

## 13. 风险与权衡

| 风险 | 处理 |
|---|---|
| 浏览器关闭后定时提醒失效 | 标注 demo 限制；预留 Web Notification API；可演进 PWA |
| localStorage 容量 / 多用户 | 按 userId 分 namespace；提供清空入口 |
| 规则引擎死板 | engine 做去重 + 优先级 + 每日上限 |
| 时区跨日 | 统一 ISO8601；按用户本地日分桶 |
| 适老化 vs 信息密度 | 提供精简 / 完整双模式，默认精简 |
